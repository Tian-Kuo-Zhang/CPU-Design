# CPU 加解密系统说明书（EES331）

## 1. 这套系统在做什么
本系统运行在 EES331 开发板上，用自定义 CPU 指令集实现一个 8 位加密/解密装置：

- 输入：`SW[7:0]`（8 位拨码开关）
- 输出：`LED[7:0]`（8 位 LED）
- 模式：  
  - `mode=0`：加密（数码管显示 `En`）  
  - `mode=1`：解密（数码管显示 `dE`）

一句话理解：  
你拨一个 8 位值进来，系统按当前模式做“加密”或“解密”，结果显示在 LED。

---

## 2. 为什么要这样设计（你关心的“能不能解回来”）
加密系统最关键是：**可逆**。  
也就是“先加密，再解密”必须回到原始输入。

为保证可逆性，本设计把每一步都选成 8 位可逆操作：

1. `+常量`（模 256 加法，可逆）
2. `LUT 查表置换`（一一映射，可逆）
3. `XOR 常量`（自逆）
4. `+常量`（可逆）
5. `LUT 查表置换`（可逆）

其中 LUT 使用的是 involution（对合）表，满足：
`LUT[LUT[x]] = x`

这使得解密时可以复用同一张表，不需要再存一张逆表。

---

## 3. 算法定义（当前实际实现）
常量定义：

- `K1 = 0x3D`
- `K2 = 0x6B`
- `K3 = 0x27`

LUT 定义：

- 文件：`crypto_lut.mem`
- 范围：`0x00 ~ 0xFF`
- 特性：`LUT[LUT[x]] = x`

### 3.3 LUT 里到底是什么
`LUT` 可以理解为一个 256 项的“替换表”（S-box 风格），每个输入字节对应唯一输出字节。  
在工程里它存放在 `crypto_lut.mem` 中，格式是每行一个 16 进制值，例如前几项：

- `LUT[0x00] = 0x86`
- `LUT[0x01] = 0x21`
- `LUT[0x02] = 0x9C`
- `LUT[0x03] = 0x40`

因为该表是对合（involution），所以还满足：

- `LUT[0x86] = 0x00`
- `LUT[0x21] = 0x01`

这就是 `LUT[LUT[x]] = x` 的直观含义。

### 3.4 LUT 是怎么“工作”的
在硬件里，LUT 表被预加载到 `data_ram[0..255]`：

- `$readmemh(\"crypto_lut.mem\", data_ram);`

程序执行 `load R1, #0x00` 时，本质动作是：

1. 取 `R1` 的低 8 位作为地址 `addr`
2. 访问 `data_ram[addr]`
3. 把读出的表值写回 `R1`

因此 `load R1, #0x00` 在算法语义上就是：

- `R1 = LUT[R1]`

这一步在加密与解密路径里都出现，是系统非线性变换的核心来源。

### 3.1 加密 `Enc(x)`
`x` 为 8 位输入：

1. `x = x + K1`
2. `x = LUT[x]`
3. `x = x XOR K2`
4. `x = x + K3`
5. `x = LUT[x]`

输出即密文 `y`。

### 3.2 解密 `Dec(y)`
`y` 为 8 位密文：

1. `y = LUT[y]`
2. `y = y - K3`
3. `y = y XOR K2`
4. `y = LUT[y]`
5. `y = y - K1`

输出回到明文 `x`。

## 4. 指令级实现（和程序文件对应）
程序文件：`program_crypto.asm` / `program_crypto.mem`

关键点：

- `load R1, #0x10`：读拨码 `SW`
- `load R2, #0x12`：读模式位 `mode`
- `beq` 分支到加密或解密路径
- `load R1, #0x00`：触发 LUT 查表（从 `data_ram` 取 `LUT[R1]`）
- `store R1, #0x11`：把结果写到 LED

IO 映射：

- `0x0010`：`SW[7:0]`（只读）
- `0x0011`：`LED[7:0]`（写）
- `0x0012`：`mode`（bit0）

---
## 5. 重点示例：输入 `10101010` 的完整加密过程
我们把二进制 `10101010` 记为十六进制 `0xAA`。

### 5.1 加密过程（`mode=0`）
初值：

- 明文 `x0 = 0xAA`（二进制 `10101010`）

步骤：

1. `x1 = x0 + 0x3D = 0xAA + 0x3D = 0xE7`  
2. `x2 = LUT[x1] = LUT[0xE7] = 0xC3`  
3. `x3 = x2 XOR 0x6B = 0xC3 XOR 0x6B = 0xA8`  
4. `x4 = x3 + 0x27 = 0xA8 + 0x27 = 0xCF`  
5. `x5 = LUT[x4] = LUT[0xCF] = 0xB2`  

最终密文：

- `y = 0xB2`（二进制 `10110010`）

---

## 6. 逆过程：把 `0xB2` 解回 `10101010`
### 6.1 解密过程（`mode=1`）
初值：

- 密文 `y0 = 0xB2`

步骤：

1. `y1 = LUT[y0] = LUT[0xB2] = 0xCF`  
2. `y2 = y1 - 0x27 = 0xCF - 0x27 = 0xA8`  
3. `y3 = y2 XOR 0x6B = 0xA8 XOR 0x6B = 0xC3`  
4. `y4 = LUT[y3] = LUT[0xC3] = 0xE7`  
5. `y5 = y4 - 0x3D = 0xE7 - 0x3D = 0xAA`  

最终解密结果：

- `x = 0xAA`（二进制 `10101010`）

与原输入完全一致。

补充观察（对应 LUT 的对合性质）：

- 加密第 4 步得到 `0xCF`，解密第 1 步 `LUT[0xB2]` 正好回到 `0xCF`  
- 解密第 4 步 `LUT[0xC3] = 0xE7`，与加密第 1 步后的值呼应  

这就是同一张 LUT 能同时服务加密与解密的原因。
